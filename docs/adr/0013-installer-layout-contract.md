# ADR 0013 — Installer Layout Contract

- **Date:** 2026-05-12
- **Status:** Accepted
- **Deciders:** operator, Claude Opus 4.7 (v5 Tier 1.2 design session)

## Context

v5.0's one-command installer is the first time mediastack writes itself onto a host filesystem in a way the project owns end-to-end. v4.x installs were hand-rolled or operator-scripted; v5.0's installer is project-authored, project-tested, and project-audited. The installer's filesystem layout is therefore both load-bearing and precedent-setting: every later step in v5 — Tier 2 dependency-install and code placement, Tier 3 first-run smoke test, Tier 4 uninstaller subcommands, the v5.0.0 audit gate — consumes the decisions in this ADR as if they were API. A vague layout contract translates into Sonnet sessions making per-task layout judgment calls; those calls then conflict at the audit gate.

The contract has five independent but composing parts: (1) which filesystem paths mediastack owns and how they separate code from data; (2) the on-disk state record the installer uses for idempotency, status, and uninstall; (3) the interactivity rules for the two distribution channels (curl|bash and git-clone-then-`sudo`); (4) the refusal logic when a host shows signs of a prior install; (5) the unprivileged-system-user model that owns the runtime. Each is documented below as a separate decision; their interactions are called out where they matter.

Decisions D1–D11 in `docs/SESSION_CONTEXT.md` are inputs to this ADR, not subjects of it. Where this ADR makes a choice not pre-determined by D1–D11, the rationale is explicit. Where it merely formalizes a decision already made there, the citation is explicit.

## Scope

In scope: installer-owned filesystem paths, the `.installer-state.json` schema and lifecycle, the curl|bash interactivity contract, existing-install detection and refusal, and the install-user model.

Out of scope and tracked elsewhere:

- **Uninstaller semantics.** Exactly what `uninstall`, `purge`, and `clean` remove or preserve is ADR 0017's subject (V5_INSTALLER_PLAN.md Step 4.1.a). ADR 0013 establishes the filesystem boundaries those subcommands operate on; the policy for which boundary each subcommand crosses is 0017's work.
- **First-run wizard UX.** The post-install handoff from "service is running" to "operator configures via web wizard" is V5_INSTALLER_PLAN.md Tier 3. ADR 0013 stops at "service is running, state file is `phase: installed`."
- **v4.x → v5.0 migration.** Explicitly out per Direction Decision D7. The refusal logic in §4 treats a v4.x install dir as `partial` (no state file present) and refuses with a message naming manual cleanup; it does not migrate.
- **Multi-distro per-family install logic** beyond Debian 12+ and Ubuntu 22.04+ per D4. New distro families (Fedora, RHEL, Arch, openSUSE) will require their own dependency-install code paths but should not require changes to ADR 0013's layout contract; if they do, that's a signal the contract is wrong.
- **Rootless Docker and alternative isolation models.** §5's docker-group membership formalizes the current project constraint per the security note there. A reconsideration is v5.1-or-later work under its own ADR.
- **Multi-user host scenarios.** mediastack is single-tenant by architecture. The installer assumes the host's role is to run mediastack; other unrelated users on the same host are not part of the contract. §5's idempotency rules tolerate the existence of unrelated users but make no commitment to coexistence beyond not stepping on their UIDs or group names.
- **Contents of `/var/lib/mediastack/`.** §1 enumerates the canonical layout per `ARCHITECTURE.md` so the audit gate knows what to expect, but the layout below the top-level directory is the backend's concern and is governed by patterns predating v5. The audit-gate purge check uses the directory itself as the unit of removal, not its enumerated subpaths.
- **The backend's mechanism for learning the data-dir path.** §1 names the contract (installer tells the backend; backend honors it via the agreed mechanism) but the implementation — systemd `Environment=` directive, `EnvironmentFile=`, command-line flag, or a config file — is V5_INSTALLER_PLAN.md Step 2.7's design work.

## Decision

### §1 — Path layout

mediastack v5.0 occupies three host filesystem locations. Each has a single owner, a single purpose, and a single lifecycle under the uninstaller subcommands (formalized in ADR 0017).

#### Code at `/opt/mediastack`

LSB-canonical for optional application software (FHS 3.0 §3.12). Contents are wholly owned by the installer: every file is either checked out from the repo, generated by a build step (`.venv/`, frontend `dist/`), or written by the installer itself (`.installer-state.json`, `POST_INSTALL.txt`). On upgrade, `/opt/mediastack` is replaced wholesale; on `uninstall`, it is removed wholesale.

Enumerated contents at v5.0 (matching the v4 repo layout the installer checks out):

- `backend/` — mediastack backend code (FastAPI; 126 endpoints across `api/`, `core/`, `health/`, `manifests/`, `infra/`, `platform/` per `ARCHITECTURE.md`).
- `frontend/` — mediastack frontend (Vue 3) source and build artifacts. Whether `node_modules/` is retained post-build is TBD per V5_INSTALLER_PLAN.md Step 2.5.a; the layout contract permits either choice.
- `catalog/` — repo-shipped catalog of managed apps.
- `manifests/` — repo-shipped manifest YAML files (55 at v4.2).
- `migrations/` — database-migration scripts (consumed by `state.db` at startup).
- `installer/` — the installer module, kept post-install for `mediastack` subcommands (uninstall, status, etc.) to import.
- `cli/`, `tools/` — repo-shipped CLI scaffolding and developer tools, retained because operator-facing `ms-*` commands depend on them.
- `.venv/` — Python virtual environment for the backend; owned `mediastack:mediastack`.
- `requirements.txt`, `pyproject.toml`, `package.json`, and other repo-shipped manifests as present.
- `.installer-state.json` — the state file. See §2.
- `POST_INSTALL.txt` — operator handoff document written after the smoke test. See V5_INSTALLER_PLAN.md Step 3.1.b.

Explicitly **not** retained in `/opt/mediastack` post-install: `.git/` (removed per V5_INSTALLER_PLAN.md Step 2.3.a — installed copies are not git working trees), repo-shipped `data/` (which is a repo-internal directory for files like `coverage_map.json` and is *not* the runtime data dir — see the boundary clarification below), and repo-shipped `tests/` (developer-only; not needed at runtime).

Final-state ownership: `mediastack:mediastack`. Modes: `0755` for directories and `0644` for files by default; `.venv/` retains `pip`'s installation defaults; `.installer-state.json` is `0640` (the file names the operator's install choices and should not be world-readable). Intermediate ownership during the install (e.g. the state file briefly written by `root` before user provisioning completes) is an implementation detail; this ADR specifies the final-state contract.

#### Data at `/var/lib/mediastack`

LSB-canonical for variable application state (FHS 3.0 §5.8). The installer creates this directory and sets its ownership and mode; the runtime layout inside is mediastack-managed per existing backend conventions (predating v5). The boundary is explicit: the installer touches `/var/lib/mediastack` itself; the backend writes the files inside it.

Enumerated by the installer:

- `/var/lib/mediastack/` — owner `mediastack:mediastack`, mode `0750`. Created if absent; left alone if present (per the data-preservation contract in V5_INSTALLER_PLAN.md Step 4.2.a).

Canonical contents at runtime, per `ARCHITECTURE.md` "Layer Overview" (host-filesystem section):

- `state.db` — SQLite database, WAL journaling mode. The companion `state.db-wal` and `state.db-shm` sidecars may also exist while the database is open. Records app registry, health, operations, and per-app instance state (34 tables at v4.2).
- `data/compose/*.yaml` — one compose fragment per installed app, generated by the backend and read by the Docker daemon. The redundant nesting (`data/` inside the data dir) is existing backend nomenclature; this ADR does not rename it.
- `.env` — secrets file. **Required mode: 0600** (called out as a Critical Contract in `ARCHITECTURE.md`). The installer never creates this file; the backend creates and owns it at first-run, and is responsible for its mode.
- `config/<app>/` — per-installed-app configuration directories; created and owned by the backend at app-install time. Container UIDs for managed apps follow the existing PUID/PGID model (mediastack-runtime concern, predates v5).

The list is forward-referential: v5.x backend changes may add files or directories under `/var/lib/mediastack/` without amending this ADR, provided the top-level directory remains the unit of contract. The audit gate (V5_INSTALLER_PLAN.md Step 4.5.a finding 3) verifies the entire `/var/lib/mediastack/` directory is removed by `purge`, not individual files — which makes the check robust against any subpaths the installer doesn't enumerate.

The installer never reads from or writes inside `/var/lib/mediastack/` during install. The state file lives in `/opt/mediastack`, not `/var/lib/mediastack`, because the state file is installer-owned, not runtime-owned, and pairs lifetimes with the code dir.

#### systemd unit at `/etc/systemd/system/mediastack.service`

Standard location for locally-installed systemd units (`systemd.unit(5)`). Owner: `root:root`. Mode: `0644`. Created by the installer; removed by `uninstall`. The unit's `User=` directive binds it to the install user per §5; `WorkingDirectory=` and the path under `ExecStart=` reference `/opt/mediastack` resolved from the `--install-dir` flag (defaulted from the state file on re-runs) and are not hardcoded in the template.

#### Boundary clarifications

Four boundaries are load-bearing and named here so Tier 2/3/4 code reads from this section rather than re-deciding:

- *Code/data boundary.* On upgrade, `/opt/mediastack` is replaced; `/var/lib/mediastack` is preserved. The installer's contract is that no operator-meaningful state lives in `/opt/mediastack` except `.installer-state.json` and `POST_INSTALL.txt`, both of which are regenerable by the installer itself.
- *Installer/runtime boundary.* The installer writes only to `/opt/mediastack`, `/etc/systemd/system/mediastack.service`, and the top-level `mkdir` of `/var/lib/mediastack`. Anything inside `/var/lib/mediastack` is the running mediastack's concern.
- *State file vs configuration.* `.installer-state.json` is installer-internal bookkeeping (see §2). It is not operator-edited configuration; operator configuration is the wizard's domain (Tier 3), and is persisted by the backend inside `/var/lib/mediastack/` per its own conventions.
- *Data-dir discovery.* The backend learns the `/var/lib/mediastack/` path from the installer, not by hardcoding. The mechanism (env var in the systemd unit, `EnvironmentFile=`, command-line flag) is V5_INSTALLER_PLAN.md Step 2.7's choice; this ADR fixes the contract that the path flows installer → state file → systemd unit → backend, and is not duplicated as a literal anywhere in `backend/`. The same `--install-dir` / `--data-dir` discipline that Core Rule 5.26 enforces in `installer/` is implicit for `backend/`, although enforcement there is the backend's existing configuration patterns rather than Rule 5.26.

#### Configurability

The defaults `/opt/mediastack` and `/var/lib/mediastack` are overridable via `--install-dir` and `--data-dir` flags (and `MEDIASTACK_INSTALL_DIR`, `MEDIASTACK_DATA_DIR` env vars) per Direction Decisions D5 and D6. Core Rule 5.26 (added in V5_INSTALLER_PLAN.md Step 1.2.b) prohibits hardcoding these paths anywhere in `installer/`; they are read from CLI args at install time, written to the state file, and read from the state file by every subsequent invocation.

### §2 — State-file schema

The state file is the canonical record of what the installer did. Without it, every other installer subcommand has to deduce mediastack's identity from filesystem grep, which is fragile (the grep accidentally matches a different mediastack-named project, or a hand-rolled v4 install) and ambiguous (a half-rolled-back install looks identical to a clean install). With it, every later invocation reads structured truth from one place.

#### Location

`<install_dir>/.installer-state.json`. With default install dir, that's `/opt/mediastack/.installer-state.json`. The dot prefix is informational: it nudges casual `ls /opt/mediastack` away from suggesting the file is operator-editable. The location pairs with the code dir, not the data dir, because the state file describes the install (which lives in code dir) and is regenerable from a fresh install (whereas data is not).

#### Format

UTF-8 JSON, 2-space indented, trailing newline, written atomically via temp-file-and-rename within the same filesystem. Final-state mode `0640` (`mediastack:mediastack`). The schema is:

```json
{
  "schema_version": 1,
  "mediastack_version": "5.0.0",
  "phase": "installed",
  "started_at": "2026-05-12T14:23:11Z",
  "completed_at": "2026-05-12T14:25:47Z",
  "install_dir": "/opt/mediastack",
  "data_dir": "/var/lib/mediastack",
  "install_user": "mediastack",
  "distro": "debian",
  "distro_version": "12",
  "port": 8080,
  "smoke_test_passed": true
}
```

#### Field definitions

| Field | Type | Required | Writer | Readers | Definition |
|---|---|---|---|---|---|
| `schema_version` | integer | always | install | all | Version of this schema. Frozen at `1` for the entire v5.0.x line. Increments on any breaking change with a paired migration function. See "Schema evolution" below. |
| `mediastack_version` | string | always | install | uninstall, status, install (re-run check) | Version of mediastack being installed. Sourced from the repo tag the installer checked out (V5_INSTALLER_PLAN.md Step 2.3.a). |
| `phase` | string enum | always | install | install (re-run check), status | Either `"installing"` or `"installed"`. Set to `"installing"` at first write; transitions to `"installed"` only after the smoke test passes. A second installer invocation finding `"installing"` interprets the install as incomplete (see §4). |
| `started_at` | ISO 8601 UTC string | always | install | status, diagnostics | Timestamp of the first write (when `/opt/mediastack` is created). |
| `completed_at` | ISO 8601 UTC string | when phase=installed | install | status, diagnostics | Timestamp of the final write (when smoke test passes). Absent or `null` while phase is `"installing"`. |
| `install_dir` | absolute path string | always | install | uninstall, status, install (re-run) | The directory where mediastack code lives. Echoes the `--install-dir` argument or its default. Self-referential (the file itself is `<install_dir>/.installer-state.json`); stored explicitly so uninstall can read it without first knowing what it is. |
| `data_dir` | absolute path string | always | install | uninstall (`purge` only), status | The directory where mediastack data lives. Echoes the `--data-dir` argument or its default. Stored so `purge` knows what to remove and so `uninstall` knows what to leave alone. |
| `install_user` | string | always | install | uninstall, status | Username of the unprivileged system user the installer created. v5.0 always writes `"mediastack"` per §5; the field exists for forward compatibility and so uninstall doesn't have to assume. |
| `distro` | string | always | install | status, diagnostics | The `ID=` field from `/etc/os-release` at install time. Matches one of the entries in `installer/SUPPORTED_DISTROS.md`. |
| `distro_version` | string | always | install | status, diagnostics | The `VERSION_ID=` field from `/etc/os-release` at install time. |
| `port` | integer | always | install | smoke test, status | The HTTP port the backend binds to. Defaults to `8080` per V5_INSTALLER_PLAN.md Step 1.4.a; the smoke test (Step 3.2.a) reads it here rather than hardcoding. A `--port` flag is not part of v5.0; the field is in the schema so adding one in v5.1+ is mechanical. |
| `smoke_test_passed` | boolean | always | install (initially false), smoke test (sets true on success) | status, diagnostics, install (re-run refusal logic) | `true` if the smoke test has passed against this install; `false` if the smoke test has not yet passed or has run and failed. Initialized to `false` at both pre-write and post-write; transitioned to `true` only by the smoke test's own write per ADR 0015 §5. Combined with `phase: "installed"`, the value distinguishes a known-good install (`true`) from an install whose pipeline completed but whose runtime readiness has not been confirmed (`false`). |

Fields outside this list are rejected by `read_state_file()` (V5_INSTALLER_PLAN.md Step 1.3.d explicitly tests "schema-validation rejects unknown fields"). Fields inside this list but missing or wrong-typed are rejected with a message naming the offending field.

#### Write lifecycle

Two writes per install, decoupling pipeline completion from runtime readiness:

1. **Pre-write.** As soon as the installer has created `<install_dir>/` (the first filesystem write of the install path beyond ephemeral temp dirs), write the state file with `phase: "installing"`, `started_at` populated, `completed_at: null`, `smoke_test_passed: false`. All other required fields are known at this point.
2. **Post-write.** After the install pipeline completes without exception (after `service_install`, before the smoke test runs), rewrite the state file with `phase: "installed"`, `completed_at` populated, `smoke_test_passed: false`. The smoke test runs against this state and updates `smoke_test_passed` in a third, smoke-test-owned write (ADR 0015 §5).

The two-field design is deliberate. "Pipeline completed without exception" and "service is reachable to a first browser visit" are genuinely two different events; coupling them into a single `phase` transition would lose the information that distinguishes "smoke ran and failed" from "smoke never ran." The state file carries strictly more diagnostic value with two fields than with one. ADR 0015 specifies the smoke test's contract and its third write; this section specifies only the first two.

Between the pre-write and the post-write, any installer crash, kernel panic, or signal-kill leaves the state file in `phase: "installing"`. A second `install.sh` run on the same host then reads this and reports the S3 in-progress message per §4. After the post-write and before a successful smoke, the state file is in `phase: "installed"` with `smoke_test_passed: false` — a state ADR 0015 §7 names and handles explicitly in the refusal logic.

#### Readers

`read_state_file()` (V5_INSTALLER_PLAN.md Step 1.3.c) is the single point of access. Callers:

- `install` subcommand on re-run: reads to determine existing-install state per §4.
- `uninstall`, `purge`, `clean` subcommands: read `install_dir`, `data_dir`, `install_user` to determine what to remove. They read first, then act.
- `status` subcommand: reads everything; pretty-prints.
- Smoke test step at install time: reads `port` to know what to probe.

No code reads the file directly; everything goes through `read_state_file()`. This keeps schema validation centralized.

#### Missing, corrupted, out-of-sync

Three failure modes:

- **Missing.** `read_state_file()` returns `None`. Callers interpret per their semantics: `install` proceeds (no prior state — see §4 state `clean` or `partial`), `uninstall` refuses with "not a v5 install — manual cleanup required" per V5_INSTALLER_PLAN.md Step 4.1.b, `status` reports "no mediastack install detected."
- **Corrupted** (file exists but doesn't parse as JSON, parses but fails schema validation, or has an unknown field). `read_state_file()` raises `StateFileCorruptedError` with a message naming the validation failure. Every caller surfaces this verbatim and refuses to proceed. `--force` does **not** override this — see §4 for rationale.
- **Out-of-sync with reality** (state file says `phase: installed` but `/opt/mediastack/.venv` is missing, or the systemd unit isn't present). The installer does not verify reality against the state file at every invocation; verification is the v5.0.0 audit gate's job (V5_INSTALLER_PLAN.md Step 4.5.a INV-2). Within normal operation, the state file is trusted as the authoritative record. If reality diverges because the operator deleted files manually, the operator is expected to use `--force` to reset.

#### Schema evolution

`schema_version: 1` is frozen for the entire v5.0.x line (5.0.0 through every patch release). No schema changes — including optional-field additions — land as patch releases. Any field change (addition, removal, rename, retype, semantic change) bumps the version. When that happens:

- The new version (e.g. `2`) ships with a `migrate_1_to_2()` function in `installer/state.py`.
- `read_state_file()` reads `schema_version` first; on `1`, dispatches through `migrate_1_to_2()` to produce a `2` in memory.
- A `schema_version` higher than the installer supports raises `StateFileNewerSchemaError`: "This state file was written by a newer installer (schema version N; this installer supports up to M). Upgrade the installer, or restore a state file from a matching version."

No migration code lands in v5.0. The version field exists so that future migration is mechanical rather than archaeological.

### §3 — curl|bash interactivity contract

mediastack ships two install paths per Direction Decision D9: a headline curl|bash one-liner (no inspection, fastest path) and a documented git-clone-then-`sudo ./install.sh` path (inspectable, recommended for cautious operators). Both run the same `install.sh`. The contract differs by whether stdin is a terminal.

#### Detection

`install.sh` checks `[ -t 0 ]` (POSIX: stdin is a terminal). The result is the *mode*. The mode determines which decisions can be prompted vs. which must be supplied as flags.

- **Interactive mode** (stdin is a TTY): the installer may prompt for ambiguous decisions. v5.0 prompts for exactly one thing: when Docker is absent and `--install-docker` is not set, prompt yes/no per Direction Decision D3. All other decisions have safe defaults.
- **Pipe mode** (stdin is not a TTY — the curl|bash case): the installer may not prompt. Decisions without safe defaults must be supplied as flags; if missing, the installer fails fast with a message naming the missing flag.

#### Decisions and their handling

| Decision | Interactive default | Pipe-mode requirement |
|---|---|---|
| Install Docker if absent (`--install-docker=yes\|no`) | Prompt if Docker is absent and flag is unset. | Flag **required** if Docker is absent. Flag is permitted but unused if Docker is present. |
| Install directory (`--install-dir=<path>`) | Default `/opt/mediastack`; no prompt. | Default `/opt/mediastack`. |
| Data directory (`--data-dir=<path>`) | Default `/var/lib/mediastack`; no prompt. | Default `/var/lib/mediastack`. |
| Force over existing install (`--force`) | Default absent; no prompt. The refusal message in §4 names `--force` as the recovery path; the operator re-runs with the flag. | Default absent; same. |
| HTTP port | Default `8080` from `installer/prereq.py`'s port-availability check; if `8080` is occupied, v5.0 fails the prereq check rather than auto-selecting a different port. | Same. |

The asymmetry is deliberate: `--install-docker` is the only decision with no defensible default. Installing Docker silently is wrong (operator should consent to system-package changes that touch firewall rules, network config, and the kernel module list). Refusing silently is also wrong (the operator may want to consent and just didn't know to). Forcing the operator to pick is the honest path. Every other decision has a sensible default that's right for the common case.

#### Fail-fast ordering

When pipe mode is detected and `--install-docker` is not provided, the installer fails **before any filesystem write, user creation, package install, or other side effect**. The check is the first thing `install.sh` does after parsing arguments and detecting pipe mode — before `detect_os()`, before `check_prereqs()`, before any state-file write, before any `apt install` of bootstrap dependencies. Operators relying on curl|bash should never find themselves with a half-created `mediastack` user, or a stray `python3` install on a host that didn't have it, because they forgot a flag.

The implementation ordering is, across `install.sh` and `installer/main.py`:

1. Parse arguments into environment variables.
2. Detect whether stdin is a TTY.
3. If pipe mode and `--install-docker` is unset → print remediation message → exit 1. **No further side effects.**
4. Detect distro (read-only: opens `/etc/os-release`).
5. Install `python3` (3.11+) and `git` via apt if absent. *This is the first system-mutating step.* The pipe-mode flag check has already passed by this point.
6. Hand off to `python3 installer/main.py "$@"`.
7. Run prerequisite checks (read-only: port-availability, disk space, sudo access, init system).
8. Provision the `mediastack` user (first action with persistent host-side effects beyond bootstrap apt installs).
9. Create `/opt/mediastack` and pre-write the state file.
10. Continue install (fetch repo, setup backend venv, build frontend, render and install systemd unit).
11. Run smoke test.
12. Post-write the state file.

Steps 1–4 are pure read/parse/dispatch. The pipe-mode-without-flag check happens at step 3, before any system-mutating step. The implementer must not rearrange steps 1–3; the ordering is part of this contract, not a heuristic.

#### Headline curl|bash command

Per Direction Decision D9 and V5_INSTALLER_PLAN.md Step 4.3.b:

```
curl -fsSL https://raw.githubusercontent.com/Nnyan/SLOP/main/install.sh | sudo bash -s -- --install-docker=yes
```

The advanced path, per the same decision:

```
git clone https://github.com/Nnyan/SLOP.git
cd mediastack
sudo ./install.sh
```

The advanced path runs in interactive mode (stdin is the operator's terminal). It prompts for the Docker decision if Docker is absent.

### §4 — Existing-install refusal logic

A second `install.sh` invocation on a host with prior install signals must refuse by default and require an explicit `--force` to overwrite. The refusal logic is a five-state machine, evaluated in precedence order. Each state has a distinct operator-facing message; the messages are part of the contract, and downstream changes need to keep them consistent.

States are detected by `installer/install.py::detect_existing_install()`, which inspects three signals: the state file, the install dir, and the systemd unit. The signals are independent; their combinations form the five states below.

| # | State | Signals | `--force` behavior | Operator message gist |
|---|---|---|---|---|
| S1 | `clean` | No state file, no install dir, no systemd unit. | N/A — proceeds. | No message (proceeds to install). |
| S2 | `installed` | State file present, validates, `phase: "installed"`. | Removes install dir and systemd unit; preserves data dir; proceeds. | "mediastack X.Y.Z is already installed at `<install_dir>` (installed YYYY-MM-DD). Re-run with `--force` to reinstall (this preserves `<data_dir>`). Or use `mediastack uninstall` to remove first." |
| S3 | `in_progress` | State file present, validates, `phase: "installing"`. | Removes install dir and systemd unit; preserves data dir; proceeds. | "A previous mediastack install was interrupted (started YYYY-MM-DD, never completed). Re-run with `--force` to clean and retry. Check `journalctl -u mediastack` for diagnostics from the previous attempt." |
| S4 | `corrupted_state` | State file present, does not validate. | **No** — `--force` does not override. | "The state file at `<path>` exists but is unreadable: `<validation error>`. This is unusual and may indicate filesystem corruption, an external editor, or a state file from a future installer version. Remove `/opt/mediastack/.installer-state.json` manually, then re-run with `--force`, accepting that state recovery is your responsibility." |
| S5 | `partial` | State file absent (or unreadable as missing), but install dir exists OR systemd unit exists. | Removes whatever is present (install dir, systemd unit); data dir is untouched; proceeds. | "`<install_dir>` exists but no mediastack v5 state file is present. This may be a v4.x install, a hand-rolled deployment, or an incomplete v5 install. Re-run with `--force` to remove and replace, or remove these manually if you want to preserve their contents." |

Precedence is strict: states are checked in order S2 → S3 → S4 → S5. If none match, the state is S1 (clean) and the install proceeds. S4 is checked before S5 so that a corrupted state file is not silently reclassified as `partial` (which would be `--force`-able and would silently overwrite a state file the installer can't verify is safe to discard).

#### Rationale for S4's unconditional non-forceability

Every other state is recoverable via `--force` because the installer knows what it's removing. In S4, the installer doesn't. The state file might describe a custom `install_dir` (e.g. `/srv/mediastack`) that `--force` wouldn't touch under the default `/opt/mediastack`; might name a `data_dir` whose contents matter; might name a user other than `mediastack` that the installer wouldn't otherwise touch. Silently overwriting a state file we can't read is a class of mistake worth one extra step from the operator. The manual cleanup is also trivial: one `rm` on a known path. The explicit "state recovery is your responsibility" phrasing places ownership of the consequences on the operator who chose to bypass the safety, not on the installer.

#### `--force` semantics summary

- In S2, S3, S5: `--force` causes the installer to remove `/opt/mediastack` (or whatever the existing `install_dir` is, when readable from the state file), remove `/etc/systemd/system/mediastack.service` if present, leave `/var/lib/mediastack` untouched, and proceed with a fresh install. The data-preservation behavior matches V5_INSTALLER_PLAN.md Step 4.2.a verbatim.
- In S4: `--force` is rejected with the S4 message. The operator must intervene manually.
- In S1 (no existing install): `--force` is a no-op.

#### Detection implementation notes

`detect_existing_install()` returns an enum-like dataclass with the state and the contextual fields needed for the message (version, dates, paths). The function is pure: it reads files, does not write. It is called once per `install` invocation, before any side effects. The output drives both the message text and the refusal-vs-proceed decision.

The function does not verify that the install dir's contents match what the state file describes; reality-checking is the audit gate's job, not the per-install runtime's. A state file claiming `phase: installed` is trusted at face value.

### §5 — Install-user model

v5.0 creates an unprivileged system user `mediastack` to own the install dir, run the backend service, and hold docker group membership for managing containers. The user is distinct from any human user on the host (per Direction Decision D2: no reuse of personal accounts, including the operator's `stack` user on the development host). The decisions in this section formalize V5_INSTALLER_PLAN.md Step 2.6.a, which the plan filed as "ADR 0013 appendix" — folded directly into §5 here, since the user attributes belong with the user-model decision.

#### Useradd shape

The exact command, on Debian/Ubuntu (the v5.0 supported distros per D4):

```
useradd \
  --system \
  --user-group \
  --no-create-home \
  --home-dir /nonexistent \
  --shell /usr/sbin/nologin \
  --comment "mediastack service account" \
  mediastack
usermod --append --groups docker mediastack
```

#### Attribute table

| Attribute | Value | Rationale |
|---|---|---|
| Username | `mediastack` | Direction Decision D2. Matches the project name; collision risk on a host that already has a `mediastack` user is handled by the idempotency contract below. |
| UID range | System UID, first available below 1000 (`useradd --system` selects from the system range per `useradd(8)`). | D2. System users are conventionally non-login service accounts; the < 1000 UID also keeps `mediastack` out of the regular-user listing of GUI user-pickers and most `getent passwd \| awk '$3 >= 1000'`-style sweeps. |
| Primary group | `mediastack` (created by `--user-group`, system GID matching the UID range). | Standard pattern for service accounts; gives the user a private group that owns the install dir without granting incidental group membership to anything else. |
| Home directory | `/nonexistent` (path string set in `/etc/passwd`; directory not created via `--no-create-home`). | The data dir `/var/lib/mediastack` is **not** the home directory. Using the data dir as `$HOME` would couple two unrelated concepts (where data lives vs. where the user "lives") and would pollute the data dir with stray dotfiles if any backend-invoked tool consults `$HOME`. `/nonexistent` is the Debian convention for service accounts that have no home; systemd, `su`, and `sudo` handle it gracefully and the unused-`$HOME` case is exactly what the backend should be doing. |
| Shell | `/usr/sbin/nologin` | Standard for service accounts; prevents interactive login as `mediastack`. systemd, `su`, and `sudo -u mediastack` continue to work for administrative purposes. |
| Comment (GECOS) | `"mediastack service account"` | Human-readable identifier in `/etc/passwd` for operators inspecting the file. |
| Supplementary groups | `docker` (added via `usermod`, not `useradd --groups`, so the addition is idempotent on second run). | Required for the user to manage Docker containers without `sudo`. Security implications below. |

#### Idempotency contract

The user-provisioning step (`installer/user.py::ensure_user()` per V5_INSTALLER_PLAN.md Step 2.6.b) is idempotent and re-entrant. On invocation:

1. `getent passwd mediastack` — if absent, run the `useradd` command above, then go to step 4.
2. If present, verify: UID is in the system range (< 1000), shell is `/usr/sbin/nologin`, home is `/nonexistent`. If any of these mismatch, raise `InstallUserMismatchError`: *"User `mediastack` exists but has unexpected attributes (UID=`<n>`, shell=`<s>`, home=`<h>`; expected system UID, `/usr/sbin/nologin`, `/nonexistent`). The installer will not modify an existing user account; remove or rename the user manually before reinstalling."*
3. If attributes match, proceed.
4. `getent group docker` — if the docker group exists and `mediastack` is not a member, run `usermod --append --groups docker mediastack`. If the docker group does not exist, raise `DockerGroupMissingError` (Docker should have been installed by V5_INSTALLER_PLAN.md Step 2.2 by this point; if the group isn't there, something is wrong upstream).

The strictness in step 2 is deliberate. A pre-existing `mediastack` user with a regular UID and login shell is almost certainly someone else's account that happens to collide with the project name. Silently bringing it under installer management — changing its shell, locking it down, granting it docker group access — would be hostile. The installer refuses and the operator decides.

#### Security note: docker group membership is root-equivalent

Granting the `mediastack` user docker group membership gives it the ability to run arbitrary code as root via Docker. The canonical demonstration is `docker run -v /:/host alpine chroot /host`, which yields a root shell on the host filesystem from any docker-group member. This is a **documented project constraint**, not a v5.0 decision:

- mediastack's architecture is host-app-orchestrates-containers; the host app must be able to start, stop, build, and inspect containers without operator intervention. The Docker socket is the only stable interface for this.
- Alternative isolation models (rootless Docker, scoped sudoers for `docker` commands, Podman, Docker socket proxies) each conflict with one or more existing mediastack requirements: host networking for the wizard endpoint and traefik integration, ports under 1024, USB device passthrough for hardware transcoding by managed apps such as Jellyfin, and persistent compose-fragment generation read directly by the Docker daemon. Each could be revisited individually, but doing so during v5.0 is scope creep.
- The trade-off is accepted because mediastack is intended for single-purpose hosts where the operator already grants mediastack root-level trust (the operator is running `sudo ./install.sh`, signed off on Docker installation via `--install-docker=yes`, and runs a product whose value proposition is "I'll manage your containers for you"). On hosts where mediastack shares the system with other untrusted users, mediastack is the wrong product.

This is formalized here so that future security audits don't treat docker group membership as an oversight. Reconsidering the constraint — for example, by adopting rootless Docker — is a v5.1-or-later project under its own ADR and is explicitly out of scope here.

#### Scoped sudoers

v5.0 does **not** install a sudoers entry for the `mediastack` user. Direction Decision D2 specifies a scoped sudoers entry "only if v5 demonstrably needs one (TBD pending Tier 2 evidence)." v5.0 has not yet identified a privileged operation the `mediastack` user needs to perform outside Docker commands. If Tier 2 surfaces such a need (e.g., for `iptables` manipulation, port-bind under 1024 without `CAP_NET_BIND_SERVICE`, or `systemctl` invocations against the mediastack unit itself from within the running service), the sudoers entry lands in v5.1 under its own ADR.

#### systemd binding

The systemd unit at `/etc/systemd/system/mediastack.service` includes `User=mediastack` and `Group=mediastack`. The unit's `WorkingDirectory=` and the path under `ExecStart=` reference values resolved from the `--install-dir` flag (defaulting to `/opt/mediastack` and recorded in the state file); they are not hardcoded in the template. `EnvironmentFile=` and other unit directives are V5_INSTALLER_PLAN.md Step 2.7's design work; this ADR specifies only the user/group binding and the principle that the data-dir path reaches the backend through the unit (per §1 boundary 4).

The unit does **not** use `DynamicUser=true`. Dynamic users are an attractive alternative for service accounts (no persistent UID, no `useradd` step) but they preclude pre-creating files owned by the user: `/opt/mediastack/.venv/` and the rest of the install dir must be owned by `mediastack` before the service starts, and dynamic-user UIDs are assigned per-service-start. The installer needs a stable UID for the file-ownership pattern, and that means a regular system user.

## What this does NOT govern

- *The backend's mechanism for learning the data-dir path* — V5_INSTALLER_PLAN.md Step 2.7. §1 fixes that the path flows installer → state file → systemd unit → backend; the unit directive used is Step 2.7's choice.

- *Multi-user hosts where mediastack coexists with unrelated services* — single-tenant assumption per project architecture.

- *ARM64 audits* — `installer/SUPPORTED_DISTROS.md` defers ARM64 explicitly; this ADR inherits that scope.

## Alternatives considered

**§1: combined `/opt/mediastack` for code and data.** Rejected. The two have different lifecycles: code is replaced on upgrade, data is preserved. Combining them either forces the uninstaller to be selective about what it removes from a single tree (fragile, easy to get wrong), or it preserves nothing on upgrade (which loses operator-meaningful state — `state.db`, `.env`, compose fragments, per-app configs). The FHS separation maps directly onto the lifecycle separation.

**§1: `/usr/local/mediastack` instead of `/opt/mediastack`.** Rejected. `/usr/local` is FHS-canonical for software the local administrator builds or installs and that may be shared by hosts on the same site; `/opt` is for self-contained third-party software. mediastack is third-party from the host's perspective and is a single-host install pattern; `/opt` is the more accurate fit. The peer projects (CasaOS, Umbrel, Runtipi) also use `/opt`-or-equivalent.

**§1: separate `/etc/mediastack/` for configuration.** Considered but rejected. v5.0 has no operator-edited configuration files outside the wizard's domain. The state file is installer-internal bookkeeping, not configuration; placing it under `/etc/` would invite operator editing. Backend secrets (`.env`) live with the rest of the runtime data per existing convention. If v5.x grows operator-edited configuration outside the wizard, that's the time to introduce `/etc/mediastack/`; v5.0 doesn't need it.

**§2: post-write only (no pre-write).** Rejected. A crashed install with no state file is indistinguishable from a clean host, except for the install dir being half-populated. The refusal message would degrade to "install dir exists, refusing" with no `started_at`, no failed phase, no diagnostic. The hybrid write costs one extra disk write and one extra field in the schema (`phase`); the diagnostic improvement is meaningful for the curl|bash use case where operators have minimal visibility into what failed.

**§2: state file in `/var/lib/mediastack/.installer-state.json`.** Rejected. The state file describes the install (code dir contents, distro, install version) and is regenerable from a fresh install. The data dir's lifecycle is "preserved across reinstalls"; placing the state file there would couple installer bookkeeping to data preservation, which is wrong. The state file is regenerable; the data is not.

**§2: schema with no version field.** Rejected. v5.0 is the first installer; v5.1+ will inevitably want to add fields. Without a version field, `read_state_file()` either rejects unknown fields strictly (forces a flag-day migration on every schema change) or accepts them permissively (loses the ability to detect a state file written by a future installer). The version field is one integer and removes the dilemma.

**§2: optional-field additions don't bump `schema_version`.** Considered briefly and rejected as internally inconsistent with the strict unknown-field rejection in `read_state_file()`. If additions don't bump the version, an older installer reading a newer file would reject the new field as "unknown" despite the version matching. Strict bump on any change makes the validation logic mechanical: version matches → all fields known. Optional-field additions cost a version bump and a trivial migration; that's the right cost.

**§2: single phase enum spanning pipeline completion and smoke success.** Rejected during ADR 0015's authoring. The implementation had already split these into two fields by Tier 2 close (commit b95016b) and the design is more honest about what the installer can and can't attest to. A single enum would either advance to `installed` only after smoke (losing the "smoke ran and failed" vs "smoke never ran" distinction) or advance after pipeline (making the enum lie about readiness). Two fields cost one boolean and gain a diagnostic surface; that's the right trade.

**§3: require `--install-docker` even in interactive mode.** Rejected. Interactive mode is the inspectable, deliberate path; an operator who runs `sudo ./install.sh` after `git clone` is in a position to answer a yes/no prompt. Requiring a flag there is busywork.

**§3: provide an unattended-install mode (`--unattended`) with opinionated defaults including `--install-docker=yes`.** Deferred to v5.1 per V5_INSTALLER_PLAN.md "Deferred to v5.1 or later." v5.0's curl|bash flag requirement is honest about the trade-off; an unattended mode papering over it should be added only after operator feedback indicates it's wanted.

**§4: treat `corrupted_state` as forceable.** Rejected. `--force` should not silently overwrite a state file the installer cannot verify is safe to discard. The state file might describe a custom `install_dir` or `data_dir` that `--force` would not address under defaults; might name a user the installer would not otherwise remove; might point to a data dir whose contents matter. One extra manual step (`rm` on a known path) is the right cost to pay for preventing a class of silent data loss.

**§4: treat `partial` as forceable only with an additional confirmation flag.** Considered. The case is: `/opt/mediastack` exists but no state file (could be v4.x, hand-rolled, or half-rolled-back). Adding a `--really-force-partial` flag would slow the legitimate case (operator deliberately replacing a hand-rolled install with v5). Plain `--force` is fine because the message names what's about to be removed and the operator has already typed `--force`; the additional friction isn't worth a marginal safety win.

**§5: home directory at `/var/lib/mediastack`.** Rejected per the rationale in §5. The data dir is not conceptually a home dir; using it as one couples two unrelated concepts and pollutes the data dir if any backend-invoked tool writes dotfiles to `$HOME`. `/nonexistent` is the cleaner separation.

**§5: home directory at `/opt/mediastack`.** Considered briefly and rejected. The install dir is owned by `mediastack`, so this would technically work, but it implies "the install dir is `mediastack`'s home," which is similarly an overload of concepts. `/nonexistent` keeps the user's identity narrowly scoped to "service account."

**§5: `DynamicUser=true` in the systemd unit.** Rejected per the rationale in §5. Dynamic users preclude pre-creating files owned by the user, which is exactly what the installer must do for `/opt/mediastack/.venv/` and the rest of the install dir.

**§5: scoped sudoers for the `mediastack` user instead of docker group membership.** Rejected for v5.0 per Direction Decision D2 ("Scoped sudoers entry only if v5 demonstrably needs one — TBD pending Tier 2 evidence"). Adding a sudoers entry now is premature optimization; v5.0 has not yet identified a privileged operation the user needs to perform outside Docker commands. If Tier 2 surfaces such a need, the sudoers entry lands in v5.1 under its own ADR.

## Status

Accepted; implemented in V5_INSTALLER_PLAN.md Tier 1.3 (`state.py`), Tier 2.6 (`user.py`), Tier 2.7 (service unit), and Tier 4.1–4.3 (uninstall and curl|bash polish). Enforced by Core Rule 5.26 (`installer-hardcoded-paths` structural check) and verified by V5_INSTALLER_PLAN.md Step 4.5 audit invariants INV-1 through INV-6.

Revisit when:

- A new supported distro family (Fedora, RHEL, Arch, openSUSE in v5.1) reveals a layout assumption that doesn't generalize — e.g., a distro whose package manager places systemd units outside `/etc/systemd/system/`, or whose docker package uses a non-`docker` group name. Such cases are amendments to this ADR, not new ADRs.
- Operator feedback indicates the `--install-docker` pipe-mode requirement is friction worth eliminating with an `--unattended` flag (v5.1+).
- The Docker group / root-equivalence trade-off in §5 changes, either because mediastack's architecture moves off the Docker socket or because rootless Docker becomes viable for mediastack's host-networking and device-passthrough requirements (v5.1+ under its own ADR).
- A schema change requires `schema_version: 2` and the first migration function in `installer/state.py`. The migration's existence is itself a signal that this ADR's "schema evolution" section in §2 should be revisited.
- The backend's runtime data layout below `/var/lib/mediastack/` changes substantially (e.g. SQLite is replaced by Postgres-on-host; `data/compose/` is renamed). The §1 enumeration is informative for the audit gate; if the audit gate's purge check starts failing on subpaths the installer doesn't know about, that's the signal to refresh §1.
