# S-45-RATCHET — File-size ratchet system

## Goal
Install a CI-enforced file-size ratchet so the codebase cannot grow files past
their current size and new files cannot exceed tiered caps. This wave only
builds the system — it does NOT remediate any existing oversize files.

## Context
Today: 236 source files (`.py` / `.vue` / `.ts`, excluding `node_modules`, venvs, dist, `.git`).
- median 256 lines, p75 456, p90 795, p95 1390, p99 2933
- 71 files ≥400 lines; 22 files >800 lines
- Existing rule (CLAUDE.md): Vue views capped at 600 — well-established but not mechanically enforced
- `frontend/src/composables/` exists but only has 2 entries — pattern under-used

Decision (2026-05-27): adopt tiered caps with CI-only enforcement via a
baseline ratchet. Mass remediation is parked in `OPTIONAL-FILE-SIZE-REMEDIATION.md`.

## Tiered caps to encode

| Category | Soft (warn) | Hard / ratchet ceiling |
|---|---|---|
| Production code: `backend/core/**`, `backend/health/**`, `backend/manifests/**`, `backend/platform/**`, `backend/infra/**`, `backend/agent/**` | 400 | **500** |
| API routers: `backend/api/**` | 600 | **800** |
| Vue views: `frontend/src/views/**.vue` | 500 | **600** (existing rule) |
| Frontend other: `frontend/src/**` excluding views | 400 | **500** |
| CLI / installer: `cli/**`, `installer/**` | 600 | **800** |
| Tests: `tests/**`, `installer/tests/**` | — | **1000** informational, does NOT fail CI |

Any baselined file (already over its category cap) may shrink but must not grow.

## Rules to follow
- This wave touches CI config, a new tool, the baseline JSON, and CLAUDE.md. It must NOT
  modify any application code, manifests, or frontend logic.
- The ratchet script must be pure-Python stdlib only (no new dependencies).
- The baseline JSON is generated by the script itself in `--snapshot` mode, then committed.
- The CI gate must integrate with the existing `ms-enforce` system if practical (see Stream B).

## Deliverables

### Stream A — Tooling + CI

#### A1. `tools/check_linecount.py`
Pure-Python (stdlib only). Modes:
- `python3 tools/check_linecount.py --snapshot` — walks the repo, writes
  `.linecount-baseline.json` with every file currently ABOVE its category hard cap
  (one entry per file: `{path, lines, category}`).
- `python3 tools/check_linecount.py` (default = check mode) — walks the repo and
  fails (exit 1) if:
  1. A file not in the baseline exceeds its category hard cap.
  2. A baselined file's line count exceeds its recorded baseline.
- `python3 tools/check_linecount.py --update-shrunk` — for any baselined file
  whose current size is *lower* than its baseline, rewrite the baseline to the
  new (smaller) value. This is the ratchet-tightening step run manually after
  intentional shrinking.

Output format on failure: list each violating file with `path:current_lines / cap_or_baseline (category)`.

File walking rules:
- Include extensions: `.py`, `.vue`, `.ts`
- Exclude paths: `**/node_modules/**`, `**/.venv/**`, `**/venv/**`, `**/dist/**`,
  `**/.git/**`, `**/__pycache__/**`, `**/build/**`, `**/.next/**`
- Category is determined by matching the path against the patterns in the
  tiered-caps table above. Patterns are evaluated in the order listed (first
  match wins) — so views match before "frontend other," API routers match before
  "production code," etc.
- Line count = `len(file.read_text(errors='ignore').splitlines())`. No trimming
  of blank lines or comments.

Tests informational-only behavior: when a tests file exceeds 1000, print a
WARNING line to stdout but do NOT fail CI.

#### A2. Generate `.linecount-baseline.json`
Run `python3 tools/check_linecount.py --snapshot` and commit the output. This is
the frozen ceiling for every existing oversize file as of this wave.

JSON shape:
```json
{
  "generated_at": "2026-05-27T...",
  "caps": {
    "production_code": 500,
    "api_routers": 800,
    "vue_views": 600,
    "frontend_other": 500,
    "cli_installer": 800,
    "tests": 1000
  },
  "files": [
    {"path": "backend/api/platform.py", "lines": 1926, "category": "api_routers"},
    {"path": "backend/manifests/executor.py", "lines": 2011, "category": "production_code"},
    ...
  ]
}
```

#### A3. CI wiring
Inspect `ms-enforce` (top-level Python entry point invoked by `.github/workflows/enforce.yml`).
- If `ms-enforce` is a dispatcher with discrete checks: add a new check that
  runs `tools/check_linecount.py` and propagates exit code. Prefer this — single CI gate.
- If integration is non-trivial: add a new step to `.github/workflows/enforce.yml`
  named "File-size ratchet" that runs `python3 tools/check_linecount.py`
  directly, after the existing ms-enforce step. Document the choice in the wave summary.

#### A4. Unit tests
`tests/test_linecount_ratchet.py` covering:
- Category resolution (each pattern matches as expected, including precedence)
- New-file-over-cap → fail
- Baselined-file-grew → fail
- Baselined-file-shrunk → pass (default mode), updates baseline (`--update-shrunk` mode)
- Tests-over-1000 → pass with WARNING to stdout

### Stream B — Documentation

#### B1. `CLAUDE.md` — add "File size limits" section
Replace the existing "Frontend architecture rule — NO business logic in view files" section's
hard-line-limit paragraph (the one stating "Hard line limit: 600 lines per new view file")
with a pointer to the new global section. Then add a new top-level section
"File size limits (ratchet-enforced)" that includes:

- The tiered caps table (verbatim from this wave)
- One paragraph on how the ratchet works:
  "New files must come in under their category hard cap. Existing oversize files
  are recorded in `.linecount-baseline.json` and may shrink but never grow.
  CI fails the PR if either rule is broken."
- "How to update the baseline" — three short procedures:
  - Intentionally shrunk a file: run `python3 tools/check_linecount.py --update-shrunk`
    and commit the new baseline.
  - Need to split a baselined file: split it, then run `--snapshot` or `--update-shrunk` to
    refresh.
  - Need to (rarely) raise a baseline: edit `.linecount-baseline.json` by hand, include
    justification in the commit message. Discouraged.
- Keep the existing "NO business logic in views" guidance — it still applies.

Do NOT modify other sections of CLAUDE.md.
